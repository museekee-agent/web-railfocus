# FlightFocus for 한국철도 — 상세 기획서

> 이 문서는 코딩 실행용 AI(저렴한 모델)에게 그대로 넘겨서 단계별로 시키기 위한 스펙 문서입니다.
> 각 섹션은 "왜 이렇게 하는지"와 "정확히 무엇을 만들어야 하는지"를 함께 적었습니다.

---

## 0. 핵심 설계 원칙

1. **"노선(선로)"과 "계통(운행 경로)"과 "열차(개별 편성)"는 서로 다른 3개의 데이터 레이어다.**
   사용자가 겪는 거의 모든 어려움(완급행, 경부1·2·3선, 천안직결선, 편성별 시간표)은 이 3개를 하나로 뭉뚱그려서 생각하기 때문에 발생함. 처음부터 분리해서 설계하면 대부분 자동으로 풀림.
2. **시간표 정확도가 물리적 정확도보다 우선.** 가감속·최고속도는 "그럴듯하게 보이는" 근사치면 충분하고, 대신 역 도착/출발 시각은 실제 시간표와 반드시 일치해야 함.
3. **MVP는 자동화보다 수동 큐레이션.** 값싼 AI가 전국 철도망을 통째로 자동 처리하려다 망가지는 것보다, 사람이 검증한 노선 몇 개를 정확히 만드는 게 훨씬 안전함. 자동화는 나중 단계.
4. **디자인은 지도 타일 없이 직접 그린다.** Mapbox/Google 스타일의 화려한 지도 대신, 행정구역 경계 geojson을 옅은 회색으로 깔고 그 위에 철도선만 그리는 방식(=FlightFocus가 실제로 쓰는 방식과 동일)이 "AI스러운 그라데이션" 없이 클로드 웹사이트 같은 플랫한 느낌을 내기 가장 쉬움.

## 1. 데이터 소스 (확정)

### 1-1. 선로 지오메트리 (geojson)
- **1순위: OpenStreetMap Overpass API**
  - 국가철도공단의 공식 전국 선로 geojson/shp는 대중 공개가 안 되어 있음(도시광역철도 역/노선 표준데이터는 있지만 일반철도 선형 원본은 없음).
  - OSM에는 `railway=rail`, `usage=main` 태그로 한국 전국 선로가 상당히 정밀하게 그려져 있음(복선 구간은 실제로 선로 2개가 별도 LineString으로 존재).
  - Overpass 쿼리 예시(실행 AI가 그대로 쓸 수 있게):
    ```
    [out:json][timeout:180];
    area["ISO3166-1"="KR"][admin_level=2]->.kr;
    (
      way["railway"="rail"](area.kr);
    );
    out geom;
    ```
  - 결과를 `osmtogeojson` 등으로 GeoJSON LineString FeatureCollection으로 변환.
- **2순위(보조/검증용): 국가철도공단 공공데이터포털**
  - `국가철도공단_도시광역철도_역사정보`(위경도 포함, XLSX) — 역 좌표 검증용.
  - `국가철도공단_도시광역철도_노선정보`, `운행정보` — 광역철도 한정이라 일반열차(무궁화·ITX·KTX)엔 부족하지만 교차검증에 사용.

### 1-2. 역 정보 (좌표, 역명, 노선 소속)
- **레일포털(KRIC, data.kric.go.kr)** — 회원가입 후 Open API 무료/무제한. 전국 역 정보(역코드, 역명, 위경도, 소속 노선) 제공.
- 위 OSM 데이터의 `railway=station`/`halt` 노드와 KRIC 역코드를 역명 기준으로 매칭(역명이 다르게 표기되는 경우가 있어 수동 매핑 테이블 필요 — 아래 4-2 참고).

### 1-3. 시간표 (역별 도착/출발 시각)
- **레일포털(KRIC) Open API** 또는 **공공데이터포털 `한국철도공사_열차운행정보`(여객열차 운행계획/운행정보)** API 둘 다 신청 필요. 단, 운행정보는 보존기간이 짧고(3개월~1일 전), 실시간성이 강해서 "정기 시간표"보다는 "실적 데이터"에 가까움 — MVP 단계에서는 이걸로 매일 자동 갱신하려 하지 말 것.
- **MVP 권장안: 수동 큐레이션.** 코레일 홈페이지의 "시간표 조회"에서 실제 열차 10~30개를 골라 역별 도착/출발 시각을 사람이 직접 표로 옮겨 적어 JSON으로 만든다. 자동 크롤링은 이용약관 이슈가 있을 수 있고, 값싼 AI가 스크래퍼를 잘못 만들 위험도 크므로 1차 버전에서는 피한다. 데이터 자동 수집·갱신 파이프라인은 Phase 3 이후 별도 검토.

## 2. 데이터 모델 (핵심 — 여기서 사용자의 모든 질문이 풀림)

3단 계층으로 나눈다.

### 레이어 A — Track Graph (물리 선로, 노선명과 무관)
철도 선로를 "정거장/분기점 = 노드, 그 사이 선로 = 엣지"인 그래프로 만든다. 노선명(경부선/전라선 등)은 여기서는 **엣지의 속성값일 뿐**이며, 실제 열차 경로 계산에는 쓰이지 않는다.

```json
// nodes.json
{ "id": "N_CHEONAN", "name": "천안", "lat": 36.8102, "lng": 127.1454, "type": "station|junction" }

// edges.json
{
  "id": "E_CHEONAN_ASAN_JANGHANG",
  "from": "N_CHEONAN",
  "to": "N_ASAN",
  "line_name": "천안직결선",
  "geometry": [[127.14,36.81],[127.10,36.79], "..."],   // OSM에서 추출한 실제 좌표열
  "length_m": 12500
}
```
- 분기점(직결선, 연결선 등: 천안직결선, 오송연결선, 부발연결선, 서정리연결선 등)도 반드시 별도 노드/엣지로 포함시켜야 4-3에서 다루는 문제(평택→아산)가 풀린다.
- 복선 구간(경부1·2·3선처럼 물리적으로 여러 가닥)은 그래프에는 그대로 다 넣되(정확도용), **화면 표시용 별도 필드**를 둔다 → 3장 참고.

### 레이어 B — Service Path (계통 = 실제 주행 경로)
"전라선 KTX", "경부선 무궁화 완행", "장항선 무궁화" 같은 **운행 계통** 단위로, 그래프 위의 엣지들을 순서대로 나열한 경로 하나를 사람이 미리 정의(큐레이션)해 둔다. 이게 "전라선 KTX가 사실은 용산부터 출발한다"는 문제의 해답이다 — 계통을 만들 때 처음부터 실제 물리 경로(용산→...→익산→...여수엑스포) 전체를 이어 붙이면 된다.

```json
// service_paths.json
{
  "id": "SP_JEOLLA_KTX",
  "name": "전라선 KTX 계통",
  "edge_sequence": ["E_YONGSAN_YEONGDEUNGPO", "E_...", "...", "E_IKSAN_YEOSU_EXPO"]
}
```
- 계통 수는 전국에 몇십 개 수준(경부선 계열, 호남선 계열, 전라선 계열, 장항선 계열, 중앙선, 경전선, 태백선 등)이므로 사람이 한 번씩 검증해서 만드는 게 충분히 현실적.
- 계통 하나를 만들 때 총 길이(길이 누적 배열)도 함께 계산해 저장 → turf.js `along()` 계산에 사용.

### 레이어 C — Train Run (개별 열차 = 완급행/편성 문제의 해답)
"열차"는 계통과 별개의 독립적인 데이터로, **"이 계통 경로 중 어디서 어디까지, 어느 역에 몇 시에 서는가"**만 갖는다. 완행이든 급행이든 같은 계통을 참조하되 정차역 목록만 다르다.

```json
// train_runs.json
{
  "train_no": "1234",
  "name": "무궁화호 1234",
  "service_path_id": "SP_GYEONGBU_LINE",
  "operating_days": ["mon","tue","wed","thu","fri","sat","sun"],
  "stops": [
    { "station_id": "N_SEOUL",   "arrival": null,     "departure": "08:00" },
    { "station_id": "N_YEONGDEUNGPO", "arrival": "08:10", "departure": "08:11" },
    { "station_id": "N_SUWON",  "arrival": "08:35", "departure": "08:36" },
    { "station_id": "N_BUSAN",  "arrival": "13:20", "departure": null }
  ]
}
```
- **완행 vs 급행**: 그냥 `stops` 배열이 길고 짧은 것 차이일 뿐, 별도 로직이 필요 없음. 급행은 필수정차역만 담긴 stops, 완행은 모든 역이 담긴 stops.
- 열차가 실제로 서지 않는 중간역이라도, 계통 경로(edge_sequence) 상에는 존재하므로 **애니메이션은 그 역을 무정차 통과하는 것처럼 자연스럽게 지나감**(가감속 없이 감속 없이 통과).
- 열차 선택 UI에서 사용자는 "노선"이 아니라 **이 Train Run 하나**를 고르는 것 → FlightFocus에서 항공편 하나를 고르는 것과 완전히 동일한 구조.

## 3. 경부1·2·3선처럼 병렬 선로가 난잡해 보이는 문제

**해결 원칙: "정확한 그래프"와 "화면에 그리는 선"을 분리한다.**

- 레이어 A(Track Graph)에는 실제 선로 가닥(1·2·3선)을 모두 정확히 넣어도 되지만, **지도에 그릴 때는 corridor(회랑) 단위로 딱 1개의 대표 선(centerline)만 그린다.**
- 방법: 전국 주요 간선을 20~30개의 "corridor" 그룹으로 미리 정의(경부선 corridor, 호남선 corridor, 전라선 corridor 등)하고, 각 corridor마다 대표 선형 1개를 손으로 고르거나(OSM에서 가장 대표적인 궤도 1가닥 선택) 또는 여러 가닥을 살짝 단순화해서 하나로 합친 선을 사람이 만든다.
- **열차 애니메이션 위치 계산은 항상 이 corridor 대표선 위에서만 이루어진다.** 실제로 1선을 타는지 2선을 타는지는 지도 확대 수준에서 의미가 없으므로 시각적으로 무시해도 무방.
- 즉 레이어 B(Service Path)의 `edge_sequence`도, 실무적으로는 "실제 선로 그래프"가 아니라 **corridor 그래프** 위에서 정의하는 것을 권장. Track Graph는 미래에 더 정밀한 버전을 만들 때를 위한 참고용으로 남겨두고, MVP는 처음부터 corridor 단위로 단순화된 그래프 하나만 만들어도 충분함 (구현 난이도를 크게 낮춤 — 값싼 AI에게 시킬 때 이 단순화가 특히 중요).

## 4. 특수 사례 처리

### 4-1. 완행/급행 (재확인)
2-C에서 이미 해결됨. 열차마다 독립적인 `stops` 배열을 가지므로, "무궁화호 1234(완행)"과 "무궁화호 1240(급행)"은 같은 `service_path_id`(경부선 계통)를 공유하되 `stops` 개수만 다른, 완전히 별개의 레코드다.

### 4-2. 역명/좌표 매칭 이슈
OSM 역 노드와 KRIC 역코드를 역명으로 자동 매칭하면 오차가 난다(예: "천안" vs "천안(순천향대)", 폐역/신설역 이슈). → `station_alias.json` 같은 수동 매핑 테이블을 만들어 큐레이션 단계에서 1회 검증. 계통이 몇십 개뿐이므로 등장하는 역도 몇백 개 수준 → 사람이 하루 정도면 검증 가능한 규모.

### 4-3. 평택→아산 (천안직결선 경유) 같은 분기 통과
2장에서 설명한 대로, **Service Path(계통)를 만들 때 edge_sequence 안에 천안직결선 엣지를 명시적으로 포함**시키면 끝난다. 특별한 로직이 필요 없고, "계통을 정확히 큐레이션했는가"의 문제일 뿐. 장항선 계통을 만들 때 실제로 "천안(경부선 진입) → 천안직결선 → 아산(장항선 합류)"처럼 정확한 물리 경로를 그대로 이어 붙이면 됨.

## 5. 가감속·속도 애니메이션 알고리즘

목표: **정차 시각/도착 시각은 100% 정확**하게, 그 사이 움직임만 "그럴듯하게" 보이도록.

1. 두 정차역 A(출발 08:00)와 B(도착 08:35) 구간이 있다고 하자.
   - 구간 거리 `D`(m)는 계통 경로의 누적 거리에서 계산.
   - 총 소요시간 `T`(초)는 시간표에서 계산 (35분 = 2100초).
2. 가속도는 열차 종별로 고정 상수 사용 (완벽한 물리 시뮬레이션 아님, 근사치):
   - 일반열차(무궁화/ITX): a ≈ 0.4 m/s²
   - KTX: a ≈ 0.3 m/s² (가속은 느리지만 최고속도가 높음)
3. **사다리꼴 속도 프로파일**: 가속 → 정속(cruise) → 감속.
   - 미지수는 정속 구간의 속도 `v` 하나뿐. 다음 식을 만족하는 `v`를 이분탐색(binary search)으로 구한다:
     ```
     accel_time = v / a
     accel_dist = v² / (2a)
     cruise_dist = D - 2 * accel_dist
     cruise_time = cruise_dist / v
     total_time(v) = 2 * accel_time + cruise_time   // 이 값이 T가 되도록 v를 찾는다
     ```
   - `v`가 열차 종별 현실적 최고속도(무궁화 ~140km/h, ITX-새마을 150km/h, KTX 305km/h)를 초과하면 그냥 그 최고값으로 클램프(clamp)하고 넘어간다. 실제 시간표에는 여유시간(padding)이 포함되는 경우가 많아 완벽히 들어맞지 않을 수 있는데, 이 경우 **cruise 구간의 시간만 남는 시간에 맞춰 늘려서 흡수**한다(즉 도착시각 정확도가 항상 최우선).
   - 두 정차역이 너무 가까워서(D가 작아서) 가속만 하다 바로 감속해야 하는 경우(삼각형 속도 프로파일) → `cruise_dist`가 음수로 나오면 삼각형 프로파일로 전환(cruise_dist=0, v를 다시 계산: `v = sqrt(a*D)`이 되도록, 그 다음 이 v로 걸리는 시간이 T보다 짧으면 나머지는 정차역에서 대기하는 것으로 처리 — 즉 정시 출발을 위해 살짝 일찍 도착해 서있는 것처럼 보여도 무방).
4. 이렇게 구한 `accel_dist / cruise_dist / decel_dist / v`로 **"경과 시간 t → 이동 거리 s(t)"** 함수를 조각별로 정의(가속 구간은 등가속도 공식, cruise는 등속, 감속은 대칭).
5. 매 프레임(`requestAnimationFrame`)마다 현재 시각으로부터 `t`를 구하고 → `s(t)` 계산 → `turf.along(corridor_geometry, cumulative_dist + s(t))`로 좌표를 얻어 마커 위치를 갱신.
6. 정차역 통과 시(무정차 통과역)는 그냥 감속/가속 없이 지나가도 되고, 아주 약한 감속만 흉내내도 됨(선택사항, MVP에서는 생략 가능).

> 이 알고리즘은 "물리적으로 완벽"할 필요가 전혀 없다. 목적은 사용자가 화면을 볼 때 "출발할 때 스르륵 가속하고, 도착할 때 스르륵 감속하는" 느낌만 주면 충분하고, 시각(t) 정확도만 절대적으로 지켜지면 된다.

## 6. 프론트엔드 기술 스택 (권장)

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Next.js (React) | 정적 export 가능, 이후 API route 확장도 쉬움 |
| 지도 렌더링 | **MapLibre GL JS** | 무료, API 키 불필요, WebGL 기반이라 열차 애니메이션이 매끄러움. Mapbox보다 라이선스 자유로움 |
| 베이스맵 | **타일 서비스를 아예 안 씀.** 대신 대한민국 시·도 경계 geojson(국가공간정보포털에서 무료 SHP 제공)을 옅은 회색(#F2F2F2류) polygon으로 직접 그리고, 그 위에 철도선(강조색 1~2가지)만 얹는다. | FlightFocus 특유의 "깨끗한 벡터 지도" 느낌을 그대로 재현 가능하고, 제3자 타일 서비스 약관/속도 이슈에서 완전히 자유로움 |
| GIS 계산 | turf.js | `along`, `length`, `nearestPointOnLine` 등 선형 보간에 필수 |
| 스타일링 | Tailwind CSS | 유틸리티 기반이라 "플랫하고 여백 넉넉한" 디자인을 그라데이션 없이 빠르게 구현 가능 |
| 폰트 | Pretendard | 한글 웹폰트 중 가장 무난하고 플랫한 느낌, 무료 |
| 데이터 저장 | **MVP는 정적 JSON 파일**(빌드 시 포함) — 별도 DB/백엔드 불필요. 나중에 시간표 자동갱신이 필요해지면 그때 가벼운 SQLite/Postgres 도입 검토 | 배포·유지보수 단순화, 값싼 AI가 다루기 쉬움 |

색상 원칙(디자인): 기본 배경/지도는 그레이스케일 1~2톤만 사용, 열차 종별로 accent color 1개씩만 배정(예: KTX = indigo, 무궁화 = orange, ITX = green) — 그 외에는 그라데이션·그림자·화려한 효과 일체 배제.

## 7. 페이지/화면 설계

### 화면 1 — 열차 선택 페이지
- 출발역/도착역 입력(또는 자동완성 검색).
- 검색 결과를 표로: `열차종별 | 열차번호 | 출발시각 | 도착시각 | 소요시간 | 정차역 수(완행/급행 구분 표시)`.
- 한 행 클릭 → 화면 2로 이동.

### 화면 2 — 포커스 화면 (FlightFocus 본체)
- 지도가 화면 대부분을 차지 (풀블리드).
- 좌측 상단 또는 하단에 미니멀 카드 오버레이: 현재 시각 기준 진행률 바, 남은 시간, 다음 정차역, 도착 예정시각.
- 지도 위 마커(열차 아이콘)가 5장의 알고리즘대로 실시간 이동.
- 지나간 구간은 옅게, 남은 구간은 진하게 표시해 진행 상황을 시각적으로 보여줌 (FlightFocus 특유의 표현).

## 8. 배포

1. GitHub 저장소 생성, Next.js 프로젝트 push.
2. Dockerfile 작성(Next.js standalone 빌드 또는 static export + nginx 서빙).
3. Dokploy에서 GitHub 저장소 연결 → 자동 빌드/배포 파이프라인 구성(push 시 자동 재배포).
4. Dokploy 내장 Traefik으로 도메인 연결 및 SSL(Let's Encrypt) 자동 발급.
5. 정적 JSON 데이터(노선/시간표)는 레포에 함께 포함시켜 빌드 타임에 번들 → 별도 서버/DB 불필요.

## 9. 단계별 로드맵 (값싼 AI에게 이 순서 그대로 시킬 것)

- **Phase 0 — 파이프라인 검증 (노선 1개만)**
  경부선 코리더 1개, 계통 1개(서울–부산), 열차 2개(완행 1 + 급행 1)만으로 전체 파이프라인(그래프 → 계통 → 열차 → 애니메이션)이 끝까지 동작하는지 검증. 여기서 데이터 모델과 가감속 알고리즘을 확정.
- **Phase 1 — 지도/애니메이션 MVP**
  화면 2(포커스 화면)를 Phase 0 데이터로 완성. 진행률 바, 도착 예정시각 등 UI 요소 포함.
- **Phase 2 — 복잡 사례 확장**
  전라선(용산 경유), 장항선(천안직결선 경유) 등 분기가 있는 계통을 추가해 4장의 특수 사례가 실제로 잘 처리되는지 검증.
- **Phase 3 — 선택 화면 + 다중 노선**
  화면 1(검색/선택) 구현, 코리더/계통 개수를 늘려 전국 주요 간선으로 확장.
- **Phase 4 — 배포 및 폴리싱**
  8장의 배포 절차 진행, 디자인 디테일(폰트, 여백, accent color) 다듬기.

## 10. 값싼 AI에게 시킬 때 주의사항

- 한 번에 "전국 철도망 자동 처리 스크립트"를 시키지 말 것. 반드시 Phase 0(노선 1개)부터 손으로 검증 가능한 단위로 쪼개서 지시.
- 데이터 모델(2장의 3-레이어 구조)을 먼저 스펙으로 고정해서 주고, 그 스키마를 벗어나지 않게 할 것.
- "정확한 선로 그래프"와 "화면 표시용 corridor"를 헷갈리지 않도록, 처음부터 MVP는 corridor 그래프 하나만 쓰라고 명시(3장 참고) — 안 그러면 복선/분기 처리하다가 복잡도가 폭발함.
- 가감속 알고리즘(5장)은 의사코드 그대로 함수 하나로 구현하게 시키고, 별도 물리 시뮬레이션을 만들지 않도록 주의.
